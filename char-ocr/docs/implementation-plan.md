# 実装計画: 手書きシンハラ文字認識(char-ocr)

調査(クラス定義 / (gid,位置)分解 / フォント融合 / データ整形)を経て確定した実装の進め方。
モデル構造の詳細は [model-design.md](./model-design.md)、(gid,位置)分解は [glyph-placement.md](./glyph-placement.md)、
ブラウザ実行は [browser-runtime.md](./browser-runtime.md)、データセットは [datasets.md](./datasets.md)。

## 要件(確定)

- **入力**: 1 文字(1 akshara)分の手書きストローク(キャンバス)
- **出力1**: ストロークに最も適合する**グリフ(視覚字形)を表示**(分類のメイン)
- **出力2**: その 1 文字を**構成パーツに分解して提示**(子音は何 / 母音記号は何)= 学習アプリの教育的価値
- 認識クラス = **視覚字形**(コードポイント列でもフォント内部グリフでもない)
- 対象 = akshara として成立する形のみ(子音単独/hal止め/母音付き/結合子音/独立母音)。
  母音記号・hal の単体は ◌ が付き不成立 = 対象外([model-design.md](./model-design.md) の「対象クラスの境界」)

## 3 段階 + 横断軸

モデルを作る軸は 3 段階。**進め順は ①→②→③**(実装の順)。

| 段階 | 内容 | 検証する問い | 注意(調査で判明) |
| --- | --- | --- | --- |
| **① ベースライン** | 454 から 13 ゴミ除去 → 丸ごと分類(~441 クラス) | 丸ごと分類で何 % 出るかの基準線 | 454 は NEXT/重子音/anusvara を欠く。SinOCR 比 80.7% カバー = 実手書きの 2 割は未知クラス |
| **② パーツ分解ヘッド** | 役割ラベルで複数ヘッド(下記) | ②で①より上がるか / 近い候補が出るか | **NEXT ヘッドは 454 では学習不能**(サンプル 2 件)→ 真価は③のデータ投入後に測る |
| **③ フォント合成** | 3 フォント画像化で 454 の穴を埋める | 合成で②の弱点(NEXT 等)が埋まるか | 母集団は二段フィルタ(描ける 827 → 実在 626)。Sinhala MN で流儀バリエーション |

**重要な依存**: ②の真価(NEXT/順序解消)は③のデータが無いと測れない(454 は ra結合のみ)。
→ ①② は 454 で**実装・配線を完成**させ(コードが動く・ONNX が出る・ヘッドが繋がる)、
**②の効果測定は③のデータ投入後**に本番。①→②→③ は順序、効果検証は③で合流。

横断軸(段階とは別レイヤー): **データ拡張** / **合成データ量・フォント数** / **ハイパラ**(lr, ヘッド損失重み λ)。

### ③ の母集団は仕様駆動チェックリストで列挙(削るだけでなく漏れを検出)

現状の「描ける827→実在626」は**フィルタ(削る)**だけ。普遍仕様の音韻マトリクス(§5-3, 子音41×母音記号16)と
Q8 特異形を**網羅チェックリスト**として重ね、各カテゴリが母集団に十分入っているかを能動検証する。
仕様側は割当 41 字すべてで機械生成できる(**「フォントに無い子音4字」はメモリの誤り。実体は未割当コードポイント
U+0DB2/0DBC/0DBE/0DBF で子音でない。割当 41 字は全部 renderable**)ので、コーパス頻度で実在ペアに絞るだけ。

**454 の穴は Sinhala Wikipedia 大規模集計(3172万トークン)で度数つきに再測定**
(`experiments/001-sinhala-wikipedia/`)。トークン視点だと文字単位の印象と逆転する:
- **454 はクラスの 21.8% しか持たないが、トークンでは 86.9% をカバー**(残り穴=13%/398万トークン)。
- **視覚クラスは度数で頭打ち**: 上位 50 字=66.6% / 100 字=82.9% / **441 字=99.3%**。
  残り 1510 クラス(L6/L7/L8 の稀な結合)は合計 0.7%。**視覚ヘッド 441 打ち切りは度数で正当**。
- **454 最大の穴は「前置コンブワ系(e/ē)」**: コンブワが base の**左**に付く字。e系 76 クラス/度数 100万・**454 内 0**、
  ē系 68/91万・454 内 1。前置系だけで未カバーの**約48%(191万トークン)**。しかも `ලෙස`(〜として)`මෙම`(この)
  `වෙනස්`(違い)等の**頻出機能語の骨格**で、稀どころか日常語。454 は ra結合(下付き)偏重で**左側修飾を1枚も学習しない**。
- 次点は **重子音/横並び結合(L7: `න්ත` `න්න` `ක්ෂ`)103万トークン・454 にゼロ**(454は全部 ra結合)。
- **o/ō は「e記号(左)+ ā記号(右)」の無変形合成**(HarfBuzz gid 実測: `කො` の右側 gid66 = `කා` の ā記号と同一、
  左側 gid74 = `මෙ` の e記号と同一)。右側は変形しておらず素グリフの足し算 → ③マルチラベルで「e@PRE」「ā@RIGHT」を
  独立に立てるのが字形と一致(o専用ラベル不要)。**Q8 の u融合(本体ごと変形)とは対照的に、e/o系は本当に足し算で済む**。

→ **③合成の優先度(度数根拠で更新)**: ①前置コンブワ系 e/ē(191万・最優先)> ②重子音 L7(103万)
  > ③Q8 角型 u(40万)。旧「Q8角型が最重要」は文字単位の見立てで、度数では前置コンブワが約5倍深刻。

| カテゴリ | 生成規則 | 454 | 備考 |
| --- | --- | --- | --- |
| L0 独立母音 / L1 子音内在a / L2 hal止め | IV / C / C+H | 有 | |
| **L3 素音節(前置コンブワ e/ē 系)** | C × {e,ē,o,ō} | **ほぼ無** | **454 最大の穴(191万tok)★最優先。`ලෙස`/`මෙම` 等の頻出語骨格。左側修飾を1枚も学習しない** |
| L3 素音節(その他) | C × DV | 一部 | u/ū 系は下記 Q8 |
| L4 鼻音化/気息 | C × VM | **無** | anusvara ★補う |
| L5 重子音 | C1+H+C2(頻出ペア) | **無** | 次点の穴(103万tok)★補う |
| L6 ra結合(rakaransaya) | C+H+ZWJ+ra | 有(56) | |
| L7 ya結合(yansaya) | C+H+ZWJ+ya | **無** | ★補う(ර 後で `yarephsinh` 化) |
| L8 repaya | ra+H+ZWJ+C | **無** | ★補う |
| **Q8 特異セル(floor 上乗せ)** | B群6字×{u,ū}・ra/ḷ×{u,ū}・ru→æ専有 | 一部 | **pixel 特異だが度数は40万 → 前置コンブワ/重子音の次。枚数を明示確保** |

- **枚数配分**: Wikipedia 度数の3層(freq≥50→48クラス/token59% / 10–49→135/29% / <10→443/12%)に **sqrt-freq + 層別 floor**。
  完全頻度比例は裾を潰す(454 と同じ失敗)、完全均等は現実分布を失う。
- **前置コンブワ floor(最優先)**: e/ē/o/ō の 4 母音記号 × 全子音は **454 がゼロに近く度数最大(191万tok)**。
  所属層に関わらず明示的に枚数を最も厚く確保する。o/ō は「e記号@PRE + ā記号@RIGHT」の足し算合成なので、
  ③ヘッドで PRE と RIGHT を独立に学習できる枚数を入れる(o専用クラスは作らない)。
- **Q8 floor**: 所属層に関わらず B群角型 u・ra/ḷ の u・ru↔ræ↔kæ の三つ巴は明示的に枚数確保(角型/標準の2モードを各々十分に)。
  ただし度数は40万で前置コンブワ(191万)・重子音(103万)の次点。pixel 特異性ゆえ floor は残すが配分の優先度はそれらの後。
- **フォント差の活用**: u の融合境界はフォント依存(Noto は A群も融合・ru は2グリフでæ専有 / MN・Sangam は A群非融合・ru は1グリフ固有合字)。
  3フォントを入れると手書きの「u 記号が独立して見える/本体に溶ける」両モードを自然にカバー。**ラベルは視覚クラスで揃え、フォント差は augmentation の分散として許容**(別クラスにしない)。
- **実装**: `akshara_table.csv` が corpus_freq・in_454・placement・3フォント融合フラグを持つので、Tier 分割と Q8 floor は CSV へのフィルタで母集団リストを出せる。
  `gen_akshara_table.py` に `q8_ushape ∈ {A,B,C}` / `is_rae` / `is_yansaya` / `is_repaya` フラグ列を追加すれば列挙・floor 付与が機械化できる。

## モデル構造

- **バックボーン**: **MobileNetV3-Small と ResNet-18 を ①で A/B**。
  - 比較を公平にするため前処理・拡張・ヘッド・損失・スケジュールは**完全共通**、差し替えはバックボーンと特徴次元のみ(ResNet18=512 / MobileNetV3-S=576)。
  - **精度・モデルサイズ(int8)・wasm 推論速度の 3 つを全部見て決める**(精度だけで選ばない)。
  - ConvNeXt は不採用(28M params は少データに過大・ロード重い)。EfficientNet-B0 は両者で不足を感じたら追加検討。
- **複数ヘッド**: バックボーンの最終分類器を `nn.Identity()` で外し、特徴ベクトルを複数の `nn.Linear` に分岐。MobileNet でも問題なく付く。
  ```
  入力 80×80 → [backbone] → [GAP] → 特徴ベクトル
      ├→ Linear 視覚ヘッド(~441)  ← 出力1
      ├→ Linear BASE ヘッド(素子音 ~54)
      ├→ Linear NEXT ヘッド(結合子音)
      └→ Linear POST ヘッド(母音記号 ~15)
  ```
- **②のラベル粒度 = 役割ラベル**(素子音 + 母音記号)。融合 gid を素子音に割り戻す(割り戻し辞書は生成可能)。
  gid を直接クラスにしない(**融合はフォント依存** = noto 5 / MN・Sangam 51。[glyph-placement.md](./glyph-placement.md))。
- **損失**: `L = L_visual + λ1·L_base + λ2·L_next + λ3·L_post`(λ はハイパラ)
- **推論**: 視覚ヘッドを第一候補(出力1)、成分ヘッドで部品提示(出力2)。矛盾時は成分整合候補へ補正可。

## データ拡張(ラスタ画像段階で統一)

**454 にストローク情報が無い(80×80 grayscale JPEG)** ため、ストローク段階拡張は不可。
454・フォント合成とも**80×80 ラスタにしてから画像変換で拡張**。推論時もストロークを同じ 80×80 ラスタに落とす(前処理一致)。

- **ライブラリ = torchvision.transforms.v2**(PyTorch 公式・追加依存なし・BSD)。
  albumentations は 2025/6 保守停止。後継 AlbumentationsX は AGPL で OSS に伝播リスクがあるため避ける。
- **入れる変形**(手書きで現実に起こるブレのみ): 平行移動 ±10% / 回転 ±12° / 拡縮 0.85–1.15 / せん断 ±8° / 弱い弾性変形 / (任意)線太さ dilate-erode 1px。
- **禁止**: 上下・左右反転(別字/ありえない入力)、大回転 >15°、強い歪み・色変換(部品配置が崩れ別字化)。
- **デフォルト**:
  ```python
  from torchvision.transforms import v2
  train_aug = v2.Compose([
      v2.RandomAffine(degrees=12, translate=(0.1, 0.1), scale=(0.85, 1.15), shear=8),
      v2.ElasticTransform(alpha=20.0, sigma=4.0),
  ])
  ```
- **強さは段階で変える**: ① は控えめ(過学習対策)/ ③ は強め(フォントは綺麗すぎ、手書きの不完全さを注入。elastic↑・線太さ・ノイズ)。
- **強さ自体がハイパラ**: 控えめスタート → 検証精度を見て調整。

## モデル軸の外で抜けていた軸(計画に含める)

### A. データ起点
- **評価データの出所**: 454 内 split だと「実手書き 2 割の未知クラス」を測れない。**SinOCR 由来 or 自作のテストセット**を別途用意(①の前提)。
- **ラベルの真偽確認**: 454 はラベルが汚い(`ද්‍රෝ` の ra消失、`ච්‍ර්` の余分 hal、13 ゴミを発見済み)。整形時に画像×ラベルの目視サンプリング。
- **クラス不均衡**: 裾(`ඞ` 1回等)が薄い。loss 重み / sampler を設計に入れるか決める。

### B. 入力(キャンバス → モデル)
- **前処理の訓練/推論一致(最優先・最大の隠れリスク)**: ストローク → 80×80 ラスタの落とし方(線幅・アンチエイリアス・余白・中心化)が訓練と推論でズレると精度が出ない。**A/B の「共通前処理」の中身そのもの**。
- **入力バリデーション**: 手書き入力には codepoint が無い。「文字未満の落書き/空白」を reject する閾値。
- **1 文字切り出し前提**: キャンバスを 1 akshara に強制する UI 前提を明示。

### C. 出力(モデル → ユーザー)
- **出力2(パーツ提示)の見せ方**: ヘッド出力を人間に分かる形(部品名 + ローマ字)にマッピングする辞書。
- **近い候補の出し方**: top-k / 成分整合補正を UI 仕様として決める。

### F. 認識後処理(結合規則バリデータ + 成分整合)

視覚ヘッド top-k と役割ヘッドの出力を、**akshara 文法**([akshara-rules.md](./akshara-rules.md))と
**普遍仕様**([../../docs/sinhala-script.md](../../docs/sinhala-script.md) §6/§7/§8)で整合チェックし、
ありえない候補を棄却/減点する。**`renderable()`(dotted-circle 検出)は文法バリデータに使えない**
(実測で `කා්`・`ක්ං`・`කිා` が dotted circle なしに描けた)ので、明示述語で判定する。

- **HARD 制約(違反候補は棄却)**: base 必須 / hal と母音記号は排他 / 母音記号は最大1個 /
  **POST に複数要素が来るのは split matra の既知4集合のみ**(`ෙ+්`=ē, `ෙ+ා`=o, `ෙ+ා+්`=ō, `ෙ+ෟ`=au。NFD で確定) /
  IV は母音記号も hal も取らない / anusvara・visarga は末尾。
- **SOFT 制約(減点)**: repaya は音節先頭 ra 限定 / rakaransaya は base 置換を伴う / yansaya は ර の後で `yarephsinh` 化。
- **Q8 補正テーブル(C13)**: 視覚=ru の æ形 & base=ර → post を u に補正(§8-2 の専有衝突)。
- **置き場所**: 制約ロジックは **TS 側 app の純関数**、制約データ(視覚クラス→部品タプル / split matra 集合 / Q8 補正表)は
  **`labels.json` に同梱**して Python(`decompose_roles` 等)で全件事前計算。
  成果物境界(model.onnx + labels.json)を壊さず、ブラウザに HarfBuzz を積まない(§E と整合)。

### D. デプロイ(ブラウザ実行。詳細は browser-runtime.md)
- **ONNX 量子化・モデルサイズ**: ResNet-18 fp32 ~45MB → int8 ~11MB。GitHub Pages の初回ロード対策。
- **推論速度**: wasm/WebGPU で 1 推論の ms。
- **既存アプリ統合**: React Router v7 SPA に新ルート追加。char-ocr(Python)→ app(TS)の成果物受け渡し。

### E. プロジェクト運営
- **成功の定義**: 「①で何 % なら②へ」の判断基準を先に決める。
- **char-ocr の成果物境界**: Python 側は「**model.onnx + labels.json を吐く**」までが責務 → app と疎結合。

## 当面のマイルストーン

1. **データ整形スクリプト**(画像不要): 454 ラベル → 13 ゴミ除去 → 役割ラベル正解 CSV + 例外リスト。`gen_akshara_table.py` の分解ロジックを再利用。
   - あわせて **`place()` の DV@NEXT バグ修正**(`රැ`/`රෑ` の母音記号が NEXT に誤判定。[model-design.md](./model-design.md) 参照)と、NEXT/POST ラベル規約を `decompose_roles` に統一。
2. **前処理の確定**: ストローク → 80×80 ラスタの落とし方を実データで詰める(訓練/推論一致)。
3. **評価データの確保**: 454 split でなく実手書き相当のテストセット。
4. **① ベースライン**: MobileNetV3-Small / ResNet-18 を A/B(丸ごと分類)→ ONNX export → onnxruntime 推論一致確認。
5. **② パーツヘッド配線** → **③ フォント合成データ**(仕様駆動チェックリストで母集団生成) → ②の効果測定。
6. **後処理(§F)**: HARD 文法制約 → 役割一致スコア → Q8 補正テーブルの3段で導入。
   HARD は 454 でも効く / 役割一致は③のデータ投入後 / Q8 補正は `ර`・`ළ` の u があれば454でも効く。
   制約データは `labels.json` に同梱、ロジックは app(TS)に置く。
