# モデル設計メモ: 結合ヘッド + 部品マルチタスク

char-ocr の文字認識モデルの設計方針。アイデア段階のメモ。
関連: データセットの詳細・label の問題は [datasets.md](./datasets.md)、ブラウザ実行環境は [browser-runtime.md](./browser-runtime.md)。

## 背景: シンハラ合成字をどう分類するか

シンハラはアブギダ。`කි`(ki)のような合成字は、視覚的には1グリフだが構造的には「子音 `ක` + 母音記号 `ි`」。
クラスの切り方に系統がある(datasets.md 参照):

- 系統1: 基本字のみ ~60クラス → 合成字を扱えず実用に足りない
- 系統2: 分割 + 結合(基本字 + 母音記号を別認識して合成) ~75クラス
- 系統3: 合成字を丸ごと1クラス → 数百クラス(本データの 454 がこれ)

## 採用アイデア: 結合ヘッド(454)をメインに、部品ヘッドをマルチタスクで併用

二者択一ではなく**併用**する。454 結合ヘッドと部品分解ヘッドは得意分野が違い、互いの弱点を補う。

| | 454 結合ヘッド | 部品分解ヘッド |
| --- | --- | --- |
| 強み | 字全体を1発で正確に当てる。**最終的に欲しい答えそのもの** | クラス少で学習安定、**部分正解・近い候補**、部品レベルのフィードバック |
| 弱み | クラス爆発、似た合成字で混同、少数クラスが弱い | 部品を当てても結合の整合は別途必要 |

### 構成案

```
入力画像 → CNN(共有エンコーダ)
              ├─ ヘッド1: 454 クラス分類     ← メインの答え
              ├─ ヘッド2: 基本字 51 クラス    ← 部品(補助 + 近さ)
              ├─ ヘッド3: 母音記号 16 クラス  ← 部品(補助 + 近さ)
              └─ (必要なら 結合子音 ්‍ර の有無 など補助ヘッド)

損失 = L(454) + λ2·L(基本字) + λ3·L(母音記号)
```

- 推論時: **454 ヘッドを第一候補**、部品ヘッドで「子音は何 / 母音記号は何」を併示
- 454 の上位候補が部品予測と矛盾したら、部品と整合する候補へ補正する後処理も可能

### なぜ併用が効くか

1. **クラス爆発の解消**: 部品は 51 + 16 ヘッドで数百合成字を表現。454 独立クラスより学習しやすい
2. **「近い候補」が出せる**(本来の狙い): `කි` を `කී` と誤っても「子音 ක は正解、母音記号だけ i↔ī で混同」と分かる。ヘッド別に確信度が出るので部分的に正しい予測ができる
3. **マルチタスク学習は正則化として効く**: 部品予測という補助タスクが共有エンコーダに字の構造を学ばせ、**454 単体より精度が上がることが多い**(タダで精度が上がりうる)
4. **学習アプリに最適**: 「子音は合ってる、母音記号の形が違うよ」という部品レベルのフィードバックは丸ごと 454 分類では出せない

## Unicode 部品分解(実データで検証済み)

454 ラベルを Unicode コードポイントに分解した結果(`tmp/decompose.py` で実測):

| 部品の種類 | 個数 | 中身 |
| --- | --- | --- |
| 基本字(子音 + 独立母音) | **51 種** | `අ ආ … ක ඛ ග … හ ළ ෆ` |
| 母音記号 + hal(pilla) | **15 種** | `ා ැ ි ී ු ූ …` + `්`(hal, U+0DCA) |
| その他の記号 | **1 種** | `ං`(anusvara, U+0D82) |
| **合計部品** | **67 種** | これで 454 クラス全部を表現できる |

合成字の構造(1ラベルあたりの部品数):
- 1 パーツ(基本字単独): 61 ラベル
- **2 パーツ(子音 + 母音記号): 311 ラベル**(大多数)
- 3〜4 パーツ(`ක්‍ර` 等の結合子音 + 母音): 80 ラベル

→ **部品ラベルは 454 ラベルから自動生成できる**(手作業不要)。前提として label の問題(ズレ・FM残骸)を先に解いておくこと。

## ソフトラベルの位置づけ

「ソフトラベル」案には2解釈がある。両立する。
- **(a) マルチラベル / マルチタスク**(部品の集合を正解とする)← 上記の主軸。確実に有効
- **(b) ソフトターゲット**(似た字に確率を分散。例 `කි` の母音を i=0.8, ī=0.2)← 似た母音記号の混同を意図的に許容する label smoothing 的手法

まず (a) を主軸に、(b) は精度チューニング段階で足す。

## 同系文字での先例: Bengali.AI コンペ(重要)

[Bengali.AI Handwritten Grapheme Classification](https://www.kaggle.com/competitions/bengaliai-cv19)(Kaggle, 2019-12〜2020-03, 2,059チーム)。
Bengali も同じアブギダで、今回の問題設定に**非常に近い**(手書きグラフェム画像、部品 = 字根 + 母音記号 + 子音記号)。

| 観点 | Bengali.AI | 今回(シンハラ) |
| --- | --- | --- |
| 文字体系 | アブギダ | アブギダ(同系) |
| 入力 | 手書きグラフェム画像 | 手書き(キャンバス) |
| 部品 | root 168 / 母音 11 / 子音 8 | 基本字 51 / 母音記号 16 |
| グラフェム総数 | 理論 ~10,000(訓練に ~1,000種) | 454(閉じている) |

### 教訓: 公式タスクは部品分解だが、優勝解法は「丸ごと分類」だった

- **公式の評価軸は部品3分類**(root / vowel_diacritic / consonant_diacritic を別々に分類)= 部品マルチタスク案そのもの
- **しかし優勝解法は部品分解を使わず**、全組合せ **14,784 (168×11×8) クラスを丸ごと分類**(EfficientNet)した方が強かった
- → 前回議論の用語で言うと、**系統3(丸ごと)が系統2(部品分解)に勝った**

### 今回への含意

- 「**454 結合ヘッドを主役**にする」方針は Bengali 勝者の選択と一致。妥当性が裏付けられた
- ただし丸ごとが勝った前提に注意: Bengali は **テストに未知の組合せ**があってもグラフェムを網羅クラス化(14,784)できたから勝てた
  - 今回のシンハラは **454 で閉じている**(未知合成字を想定しない)なら、丸ごと 454 で十分戦える
  - 逆に「**将来 未知の合成字も認識したい**」なら部品分解が有利(組合せに開かれている)
- 部品ヘッドは「精度で勝つため」ではなく、**今回固有の価値(近い候補・部品レベルのフィードバック = 学習アプリ用途)**のために併用する、という位置づけが明確になった

## 進め方(段階導入)

いきなり3ヘッド + 損失の重み(λ)調整から始めると、何が効いているか切り分けられない。
ML の鉄則「1つずつ足して効果を測る」に従う。ML 学習目的にも合う順序。

1. **まず 454 ヘッド単体**で動かす(ベースライン確立。HuggingFace の ResNet-18 と比較可能)
2. **次に部品ヘッドを追加**し、精度が上がるか・近い候補が出せるかを検証
3. うまくいけば本採用。(b) ソフトターゲットは最後に検討

## 前提となる残タスク(datasets.md より)

- label の 456 vs 454 ズレの確定(部品分解の前に必須。元ラベルが正しくないと部品も狂う)
- 壊れ4ラベル(FM Abhaya 残骸 426/450/453/454)の正字確定
- 学習自体(454 整数ラベル)は上記と並行で先行できる
