# モデル設計メモ: 成分分析ヘッド + 合成ヘッド

char-ocr の手書き文字認識モデルの設計方針。実測に基づく確定版。
関連: データセットの詳細は [datasets.md](./datasets.md)、結合ルールは [akshara-rules.md](./akshara-rules.md)、
部品分解の構造は [decomposition-table.md](./decomposition-table.md)、ブラウザ実行は [browser-runtime.md](./browser-runtime.md)。

## 要件(確定)

- **入力**: 1 文字(1 akshara)分の一連のストローク(手書きキャンバス)
- **出力1**: そのストロークに最も適合する**グリフ(視覚字形)を表示** ← 分類のメインの答え
- **出力2**: その 1 文字を**構成パーツに分解して提示**(子音は何 / 母音記号は何)← 学習アプリの教育的価値

### 対象クラスの境界(akshara として成立する形だけ)

クラスは「akshara として成立する = 子音/独立母音を土台に持つ形」に限る。実測(HarfBuzz)で確認:

| 入力 | 表示 | 対象 |
| --- | --- | --- |
| `ක`(子音単独)/ `ක්`(hal 止め)/ `කි`(母音付き)/ `ක්‍ර`(結合)/ `අ`(独立母音) | 正規の字形 | **○** |
| `ි` `්` など母音記号・hal の**単体** | `◌ි` `◌්`(◌ = U+25CC dotted circle が付く) | **✕** |

- 母音記号・hal は定義上 **dependent(従属)記号**で、子音にぶら下がってこそ成立する。**文章中に単独で現れない**(コーパスにもトークンとして出ない)。
- 単独で shape すると **◌(点線丸)= 「土台の子音が無い」警告マーカー**が付く → 表示が成立しない → 対象外。
- `ක්‍`(ka+hal+ZWJ で止め)のような「未完成」入力は、表示すると **`ක්` と同形**(ZWJ は宙ぶらりんだと幅ゼロの space に化けて消える)。よって ZWJ 除去で `ක්` クラスに畳まれ、別クラスは不要。
- **入力バリデーション**: shape 結果に `.notdef`(gid=0)・`uni25CC`・`space` 以外の実体グリフが 1 つも無ければ「文字として不成立」と判定できる。

## 大方針: 認識のクラスは「視覚的に同じ形 = 同じクラス」

コードポイント列でもフォント内部グリフでもなく、**視覚字形**で切る。
- 同じ形なら、裏のエンコード(ZWJ 有無・split matra 分解形)が違っても同一クラス。
- 正規化 = NFC + ZWJ/ZWNJ 除去。これで「視覚的同形」のキーになる。
- 実測(SinOCR): エンコード正規化で畳めるのは僅か(620→614)。手書き正解テキストは既にほぼ視覚クラス単位。

## なぜ単一の分類器ではダメか(実測で確定)

| 方式 | 結果 |
| --- | --- |
| akshara 丸ごと 1 クラス | 614 クラスだが裾が薄い(`කෞ` 2 回等)。学習困難 + 未知組合せに弱い |
| 素朴なマルチラベル(部品の有無 0/1) | **86 字が復元不能**。`න්ම`/`ම්න`(順序)、`ත්`/`ත්ත`(個数)が同集合に潰れる |
| 完全な系列予測(CTC) | 復元できるが重い。学習アプリには過剰 |

→ **構造化マルチタスク**(akshara の文法軸ごとにヘッドを立てる)が現実解。

### 重要: 順序・個数依存は「結合子音(base 複数)」だけの局所問題

実測で切り分け済み:
- **base 1 個の文字(token の 90%)= マルチラベルで完全可逆(衝突 0)**。順序を気にしなくてよい。
- 順序/個数の衝突 41 組は**全て base 複数(結合子音、token の 10%)**。`ප්ර`/`ර්ප`(順序)、`ක්`/`ක්ක`(重子音=個数)。
- → **役割分担**: 単独 base は成分分析ヘッドが分解(可逆)、結合子音は合成/視覚ヘッドが丸ごと担当。

## アイデア: 位置ベース(BASE/NEXT/POST)の役割ヘッド構造

HarfBuzz 実シェーピングで akshara を `(部品, 位置)` に分解すると、位置は実質 **3 値**に集約される
(詳細・検証は [glyph-placement.md](./glyph-placement.md))。これを使ったクラス設計案。

### 実測した位置と部品の分布(SinOCR)

| 位置 | token% | 何が来るか | 位置で役割が変わるか |
| --- | --- | --- | --- |
| **BASE** | 72.0% | 主役の基底字(子音/独立母音、hal 付き含む) | — |
| **NEXT** | 6.5% | 結合子音の 2 番目の子音(adv>0 = 横に並ぶ) | 子音は BASE/NEXT 両方に出る |
| **POST** | 21.4% | adv=0 で base に貼り付く: 母音記号(ා ෙ ැ)・anusvara ං・rakaaransaya(`ක්‍ර` の ර) | POST 固定(位置一意) |

- 部品 gid 197 種のうち **複数位置に出る 76 個は全部子音系**で、出る位置は例外なく `{BASE, NEXT}`。
- **母音記号は常に POST**(位置が動かない → 位置ラベル不要)。
- → 「位置で意味が変わる」のは**子音だけ**で、しかも **BASE か NEXT かの 2 値**に尽きる。

### 構造図

```
                        ┌─────────────────────────────────────────┐
入力画像(1 akshara) →   │      CNN 共有エンコーダ                   │
                        └─────────────────────────────────────────┘
                              │            │              │
                    ┌─────────┘     ┌──────┘        ┌──────┘
                    ▼               ▼               ▼
              【BASE ヘッド】   【NEXT ヘッド】   【POST ヘッド】
              主役の子音/母音字  結合の2番目子音    母音記号 + anusvara
              単一分類(~60)    なし or 子音(~60) なし or 記号(~15)
                    │               │               │
                    └───── 位置=順序を保持 ──────────┘
                       BASE=先頭, NEXT=後続 で
                       න්ම(na@BASE,ma@NEXT) と
                       ම්න(ma@BASE,na@NEXT) を区別

出力1(グリフ表示): 3 ヘッドの予測を組み立てて akshara を再構成
出力2(部品提示):   各ヘッドの予測がそのまま「子音は何/母音記号は何」
```

### この案の効きどころ

| 論点 | この案での解 |
| --- | --- |
| クラスが多すぎる(626) | 丸ごと分類をやめ、BASE~60 × NEXT~60 × POST~15 の**役割別小語彙**に分解 |
| 順序問題(`න්ම`/`ම්න`) | **BASE/NEXT の 2 値で解決**(どちらの子音が先頭か)。検証で衝突ゼロ |
| split matra(`කො`=ෙ+ා) | 母音記号は POST 固定。NFD 分解とグリフ配置が一致するので素直に POST へ |
| 位置情報の取得 | コードポイントに無い → HarfBuzz の返却順(reorder 済み視覚順)と advance で BASE/NEXT/POST を機械判定 |

### 未決定(粒度の選択)

BASE/NEXT の子音語彙を、フォントの融合 gid のまま(`ක` `කි` `කු` 別 = ~190)にするか、
素子音+母音記号に割り戻す(子音 ~60 + 母音記号を POST に寄せる)かは未確定。
割り戻し辞書は生成可能(`(子音 × {なし/hal/母音記号})` の shape から、確認済み)。

## 採用構成: 成分分析ヘッド + 合成ヘッド(旧案・参考)

```
入力画像 → CNN(共有エンコーダ)
   ├─ 合成ヘッド(視覚クラス分類)      ← メインの答え =「最適グリフ表示」(出力1)
   │     クラス数は採用カバレッジで選ぶ(下表)
   ├─ 成分: base ヘッド(子音/独立母音)  単一分類            ← 部品(出力2)
   ├─ 成分: 母音記号ヘッド(split 分解後) マルチラベル(素パーツ)← 部品(出力2)
   ├─ 成分: hal 有無                    2 クラス(子音のみか)
   └─ 成分: 結合フラグ(ra / ya 結合)    有無

損失 = L(視覚) + λ1·L(base) + λ2·L(母音記号) + λ3·L(hal) + λ4·L(結合)
推論: 視覚ヘッドを第一候補(出力1)、成分ヘッドで「子音は何/母音記号は何」を併示(出力2)
      視覚候補が成分予測と矛盾したら成分整合候補へ補正する後処理も可能
```

### なぜこの形か(実測の裏付け)

| 設計判断 | 根拠(実測) |
| --- | --- |
| 視覚ヘッドをメインに | 「最適グリフ表示」が要件の主目的。Bengali.AI でも丸ごと分類が勝った |
| base/母音記号を別ヘッドに | 裾の `කෞ`(2 回)も「ක(672)+ ෞ」に分ければ部品サンプルが桁違い。組合せ爆発を緩和 |
| 母音記号は split 分解後の素パーツ | `ෝ`=ෙ+ා+් とフォント自身が分解。稀な複合記号が頻出パーツに還元され救われる |
| base は単一分類でよい | base 1 個が token の 90%。結合子音(10%)は合成/視覚ヘッドに任せる |
| hal/結合を別軸に | 順序・個数の衝突(`ත්`/`ත්ත`)を避けるため軸を分離して明示的に持つ |

## クラス数: 段階的に決める(model-design 進め方に沿う)

視覚ヘッドのクラス数は学習画像が確保できる粒度で選ぶ(SinOCR コーパス頻度で測定):

| しきい値 | 視覚クラス数 | カバー率 |
| --- | --- | --- |
| freq≥2(裾の固有名詞のみ除外) | 441 | 98.4% |
| **freq≥5(各クラス最低 5 サンプル)** | **278** | 94.2% |
| freq≥10 | 183 | 88.4% |

※ ただしこれは SinOCR テキストの言語頻度。**実学習データ(454 画像)とは母集団が違う**ことに注意(下記)。

## 学習データの整形(検証済み・ここが実務のボトルネック)

実画像があるのは Kaggle「Sinhala Letter 454」(folder 1..454)。各 folder のラベルから
各ヘッドの正解タプル `(視覚クラス, base, 母音記号集合, hal, ra結合)` を自動生成できるか検証した:

- **441 / 454 は NFD 分解で完全自動整形できる** ✓(例: `කෝ`→ base=[ක] dv=[ෙ,ා] hal=T ra=F)
- **13 件が例外**(自動整形不可、手当てが必要):
  - **母音記号単独 9 件**(`ෝ ෑ ැ ා ෟ ෘ ෲ ෛ ෙ`): base が無く結合ルール上 akshara でない。
    → 画像確認要。「母音記号の形だけ」の画像なら**母音記号ヘッド専用データ**として活用可(視覚ヘッドには入れない)
  - **FM Abhaya 残骸 4 件**(`COda HQ H Hq` = idx426/450/453/454): ラテン残骸。画像から正字を人手特定
- 454 範囲で自動生成される成分語彙は小さい: **base 51 種 / 母音記号 8 素パーツ(`ා ැ ෑ ි ී ු ූ ෙ`)/ ra結合 80 ラベル**

### 454 データの限界(将来拡張時の注意)

- 454 の結合子音 80 個は**全て ra 結合(`X්‍ර`)1 パターン**。**重子音(`න්න`)・ya結合(`ක්‍ය`)・順序反転は 1 枚も無い**。
- → 454 で学習する限り**順序/個数依存は実質発生しない**(成分ヘッドが素直に効く)。
- 逆に SinOCR 実手書きで頻出の重子音・ya結合を認識したいなら 454 では不足
  (454 は SinOCR akshara の 80.7% のみカバー)。**フォント合成データ**(Noto は全 76 部品を描けるので
  教科書全字の合成生成が可能)等でデータ追加が必要。

## 進め方(段階導入: 1 つずつ足して効果を測る)

1. **まず視覚ヘッド単体**でベースライン(454 → freq で 278 等。HuggingFace ResNet-18 と比較可能)
2. **成分ヘッド(base / 母音記号 / hal / ra)を追加**し、精度 UP・「近い候補」が出るか検証
3. うまくいけば本採用。ソフトターゲット(似た母音記号に確率分散)は最後に検討

## 同系文字の先例: Bengali.AI コンペ

[Bengali.AI Handwritten Grapheme Classification](https://www.kaggle.com/competitions/bengaliai-cv19)(2019-20)。
同じアブギダで問題設定が近い(手書きグラフェム = root + 母音記号 + 子音記号)。

- 公式タスクは部品 3 分類(= 成分ヘッド案そのもの)だが、**優勝解法は全組合せ 14,784 を丸ごと分類**した方が強かった。
- 含意: 「視覚ヘッドを主役」は妥当。ただし丸ごとが勝てたのは**テスト未知組合せもグラフェム網羅クラス化できた**から。
- 今回は 454 が閉じている(未知合成字を想定しない)なら丸ごとで十分戦える。
  成分ヘッドは精度のためというより**今回固有の価値(近い候補・部品レベルのフィードバック = 学習アプリ用途)**のために併用。
