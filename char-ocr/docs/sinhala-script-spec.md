## 前提(ルート docs の要点だけ再掲)

hal(U+0DCA)は文脈で4通りに化ける(詳細はルート docs §2):

- **2a 単純 virama**(`ක්`): 子音 + 母音殺し。子音ごとにフック形は違うが通常の合字。
- **2b split母音記号の中の hal**(`ෝ`=`ෙ+ා+්`): ō/ē の構成部品。**virama と同一字形**で特殊ではない。
- **2c ZWJ で要求する3結合形**: repaya(上付き)/ yansaya(後置)/ rakaransaya(下付き)。**これが本当の特殊形**。
- **2d touching**(ZWJ+Halant 順): 密着連結。古典用。

配置は HarfBuzz の glyph 名で機械判定できる(`<C>rasinh`=rakaransaya, `yapostsinh`=yansaya,
`repaya`=repaya, `<C>halantsinh`=virama 形)。

454 は「画素特異性」では rakaransaya 中心で妥当だが、**実利用度数で見ると最大の穴は前置コンブワ系(e/ē)**:
- **e系 76クラス/度数100万・454内0、ē系 68/91万・454内1**。前置コンブワだけで未カバー約48%(191万tok)。

  「左に記号が出る」配置を ①視覚クラス/③`@PRE` で当てる訓練が不足する。
- **o/ō は前置コンブワ(e記号@PRE)+ ā記号(@RIGHT)の無変形合成**(HarfBuzz gid 実測: `කො` の右側 gid66 =

## クラス設計・補助ラベルへの含意(結論)

- **分解ラベル(コードポイント由来)は規格と一致しており正しい**。split母音記号の hal を post に含めるのも NFD 準拠
  (`scripts/gen_training_labels.py` の `decompose_roles`)。
- **「hal の帽子が特殊」という補助ラベルは不要**。virama 形と ō の hal は同一字形で、画素的に区別する根拠がない。
- **画素から予測する都合で本当に効く補助フラグは「rakaransaya(下付き ra結合)か否か」**。
  base 子音のグリフ自体が置換される=画素が通常 base と大きく異なるため、
  視覚クラスとパーツ予測の橋渡しに有効。`conjunct` 列(子音2つ以上)とは別概念で、`<C>rasinh` 検出で立てられる。
- repaya/yansaya は 454 に無いので**現データではフラグ不要**。③フォント合成で語彙拡張する場合に追加検討
  (その時も本書の判定キーで機械列挙できる)。
- 列挙・フラグ付与は HarfBuzz glyph 名(`rasinh`/`yapostsinh`/`repaya`/`halantsinh`)で機械化でき、
  `scripts/gen_akshara_table.py`(配置 + 融合フラグ出力)に統合可能。

## Q8(子音で字形が変わる付加記号)のラベル含意 ― ②素に畳む / ③視覚形で割る

普遍仕様 [../../docs/sinhala-script.md](../../docs/sinhala-script.md) §8 の字形変化が、ラベル設計に直接効く。
ラベルは音(母音名)でなく「**メイングリフ(素の土台)+ 修飾パーツ(画素上の装飾)の足し算**」で切る。
HarfBuzz 実測で確定した事実とその扱い(原則は [model-design.md](./model-design.md) の確定 3 ヘッド):

### u(◌ු)/ ū(◌ූ)の3パターンと各ヘッドでの扱い

| パターン | 対象 | ①文字全体 | ②メイングリフ | ③修飾パーツ(視覚形) |
| --- | --- | --- | --- | --- |
| A 下垂フック | 残り33字(`දු` `පු` 等) | 別クラス | 素 `ද` 等に畳む | **`u_下垂`** |
| B 角型(k-shape) | 6字 ක ග ඟ ත භ ශ | 別クラス | 素 `ක` 等に畳む(角型化しても素) | **`u_角型`** |
| C 特殊合字 | ර ළ | 別クラス | 素 `ර` `ළ` に畳む | **`u_合字`** |

- **③ は視覚形で割る**(`u_下垂`/`u_角型`/`u_合字`)。音なら畳むが、③ を「音でなく画素上の修飾」と定義したので
  視覚形で割るのが整合。② は逆に**素の土台に畳む**(B群で `ක` が角型化しても ② は `ක`)。視覚差は①が吸収。
- **視覚形ラベルは画素 metric で自動判定できない**(実測:増分の面積比・縦重心では `ද`(丸い下垂)と
  `ක`(角型)が分離できず、`ග` `ත` がグラデーションで中間に並ぶ)。Noto の glyph 名も `කු`/`දු` とも
  `<C>uvowelsinh` 1 グリフで**名前から判別不能**。→ **base コードポイント集合で決定的に振る**のが唯一確実:
  `base∈{ක ග ඟ ත භ ශ}→u_角型` / `base∈{ර ළ}→u_合字` / それ以外`→u_下垂`(フォント非依存・universal 仕様)。
- B群の角型化は**程度にグラデーション**がある(境界字 `ග` `ත` は弱め)。ラベルは集合で離散に割り切るが、
  ③フォント合成で**角型サンプルを厚く・複数フォントで**確保すべき最重要カテゴリ([[sinhala-synthetic-data-population]])。

### ර+u が æ 記号を専有する衝突(POST ヘッドの唯一の例外)

- 実測:`රු`(ru)= `rasinh#56 + aevowelsignsinh#67`。この **#67 はあらゆる `Cæ` の æ 記号と同一 gid**(`රූ` も `Cǣ` と #68 共有)。
- `aevowelsignsinh#67` を持つ akshara は 48 件、うち 47 件が役割 post=æ、**`රු` 1 件だけ post=u**(corpus_freq=145・454 在)。
- **akshara 丸ごと(視覚ヘッド)では衝突しない**(base=ra で一意)。衝突するのは POST 部品ヘッドのレベルだけ。
- → POST ヘッドを base 非依存にせず、`base=ර`(ū では `ළ`)& #67/#68 → post=u を**決定的後処理テーブル**で確定。
  `රැ`(ræ)/`රෑ`(rǣ)は ra 専用 #545/#546 に退避するので衝突しない(æ のまま)。

### ラベル検証で重点サンプルすべき字(Q8 由来)

- C群+専用形: `රු රූ ළු ළූ` / `රැ රෑ`。とくに **`රු` と `කැ`(kæ)を並べて** ③ パーツが画素(同一 #67 グリフ)と
  食い違う実例を目視(`රු`→`u_合字`、`කැ`→æ記号。同じ #67 画素から base 条件で撃ち分ける)。
- B群6字 × u/ū: `කු ගු ඟු තු භු ශු`(角型)を A群代表 `දු පු` と並べ、① は別クラス・② は素の土台に畳め・
  ③ は `u_角型`/`u_下垂` に視覚形で割れているか三面確認。境界字 `ගු` `තු` は角型が弱い点も確認。
- `tmp/akshara_table.csv` の `decomp_gid_pos` 列を `LIKE '%67:aevowelsignsinh%'` 検索で 48 件を機械列挙し、
  (視覚クラス, base, post) の三つ組が ru/rū だけ post=u になっていることを点検(duckdb で確認済み)。
